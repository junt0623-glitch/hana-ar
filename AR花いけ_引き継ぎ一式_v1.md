# AR花いけアプリ「(仮称) 花をいける」引き継ぎ一式 v1.0

作成: Claude Fable 5（設計担当）／2026-07-10
発注者: Junt（美術館学芸員・中国陶磁専門）
本書は Opus 4.8（実装）、Sonnet 4.6（素材整備・テスト）、Haiku 4.5（文言）への引き継ぎ文書である。

---

## 0. プロジェクト概要

美術館の展示作品（中国陶磁の花器）にスマートフォンをかざすと、器に花が生けられているように見えるWebアプリ。来館者は作品脇のQRコードを読み取り、カメラ画面上で季節・様式別の花を選んで配置し、合成写真を保存できる。

## 1. 確定事項（決定ログ）

| 項目 | 決定内容 |
|---|---|
| 実装方式 | 案A（手動配置型）→ 案B（画像トラッキングWebAR）の段階方式 |
| 季節区分 | 四季（4区分）。将来の細分化に耐えるデータ構造とする |
| 花の様式 | 真・行・草（生花三体）／文人花（『瓶史』考証ベース）／盆栽／盛物 |
| 花の表現 | 実物写真（透過切り抜き） |
| 素材調達 | ①華道家連携撮影＋②ストックフォト＋③AI画像生成の3方式併用 |
| 保存機能 | 搭載する（canvas合成→端末ダウンロード、サーバー送信なし） |
| 配布形態 | 単一HTML・依存ゼロ・オフライン動作・GitHub Pages手動デプロイ |

## 2. アーキテクチャ（Phase 1）

```
hana-ar.html（単一ファイル）
├─ QRエントリ層    URL: .../hana-ar.html?work=W001
├─ 作品レジストリ   works[]（ファイル内埋め込みJSON）
├─ 花レジストリ     flowers[]（透過WebPをbase64埋め込み）
├─ カメラ層        getUserMedia（背面カメラ、facingMode: environment）
├─ 配置エンジン     placementEngine（抽象インターフェース）
│    └─ ManualPlacement（Phase 1: ドラッグ・ピンチ拡縮・回転）
├─ 合成層          canvas合成 → toBlob → aタグdownload
└─ 状態層          localStorage（設定・最終選択の記憶）＋JSONバックアップ
```

### Phase 2 移行設計
- `placementEngine` に `TrackingPlacement`（MindAR.js 画像ターゲット）を追加実装して差し替える。
- 花データ・作品レジストリ・UI・QRのURL形式は Phase 1/2 共通。**展示室のQR掲示物は刷り直し不要**。
- Phase 2 で CDN 依存が発生する点は原則からの例外として承認済み。オフラインフォールバックとして手動配置モードを残す。

### 技術的注意（Opus 4.8 向け）
- iOS Safari: getUserMedia は HTTPS 必須（GitHub Pages で満たす）。カメラ起動はユーザー操作起点で行う。
- canvas 合成時、カメラ映像は `drawImage(video, ...)` で取得。CSS変形と描画座標のズレに注意（devicePixelRatio 対応）。
- ピンチ・回転は Pointer Events で自前実装（依存ゼロ原則）。キャプション工房の画像スケール／ドラッグ実装を移植可。
- ネイティブダイアログ禁止。確認はページ内モーダル（既存パターン踏襲）。
- 花画像は季節単位で遅延デコード（`<img decoding="async">` ＋ 選択時に初回描画）し、初期表示を軽くする。

## 3. データスキーマ

```json
{
  "meta": { "version": "1.0", "generated": "2026-07-10" },
  "works": [
    {
      "id": "W001",
      "title": "青花梅瓶",
      "vessel": "meiping",
      "mouthWidth": "narrow",
      "styles": ["gyo", "so", "bunjin"],
      "note": "細口。一枝ものが適合"
    }
  ],
  "flowers": [
    {
      "id": "F012",
      "name": "梅",
      "reading": "うめ",
      "season": "winter",
      "styles": ["shin", "gyo", "so", "bunjin"],
      "vesselFit": ["narrow", "medium"],
      "source": "1",
      "credit": "撮影協力: ○○流",
      "anchor": { "x": 0.5, "y": 0.97 },
      "img": "data:image/webp;base64,..."
    }
  ]
}
```

- `mouthWidth` / `vesselFit`: narrow / medium / wide / flat の4値。適合しない組み合わせはUIに表示しない。
- `styles`: shin（真）/ gyo（行）/ so（草）/ bunjin（文人花）/ bonsai（盆栽）/ morimono（盛物）。
- `season`: spring / summer / autumn / winter。跨ぎは配列可（例: 梅 = ["winter","spring"]）。
- `anchor`: 画像内の「留め」（茎の束の基準点）の相対座標。配置エンジンがこの点を器の口に合わせる。
- `source`: 調達区分 1=連携撮影 / 2=ストック / 3=AI生成。3 は考証チェック通過済みのみ登録可。

## 4. 画面フロー・UI仕様

1. **QR読取** → `?work=W001` 付きURLが開く
2. **作品情報画面**（1画面）: 作品名・器種・「カメラをはじめる」ボタン・プライバシー注記（画像はサーバーに送信されません）
3. **カメラ画面**:
   - 背景: カメラ映像（全画面）
   - 下部: 季節タブ（春夏秋冬）× 様式タブ（適合様式のみ表示）
   - 花サムネイルをタップ → 映像上に花が出現（anchor が画面中央下寄りに来る初期配置）
   - ドラッグ移動／ピンチ拡縮／二指回転
   - シャッターボタン → canvas合成 → ダウンロード
4. **文言トーン**: 来館者向け。敬体、専門用語には短い補足（Haiku 4.5 担当、Junt 監修）

## 5. 素材仕様

| 項目 | 仕様 |
|---|---|
| 形式 | WebP（透過）。切り抜き元はPNG可、最終はWebP |
| 解像度 | 縦1200px基準（長辺上限1600px） |
| 容量 | 1点 150〜250KB。全体で8MB以内を目安。超過時は季節別ファイル分割へ移行 |
| 切り抜き | **花器を含めない**。下端は「留め」（茎の束）で終端。器の口から上のみ |
| 撮影条件（①） | 無地背景（グレー推奨）、実器に生けた状態で撮影し口から上を切り出す。順光・影を抑える |
| 命名規則 | `{season}-{style}-{name}.webp` 例: `wi-bunjin-robai.webp` |
| アンカー | 留め位置の相対座標をJSONに記録（切り抜き後に計測） |

## 6. 素材調達区分と運用ルール

| 区分 | 対象 | ルール |
|---|---|---|
| ① 連携撮影 | 真・行・草の代表構成、文人花全般（様式的正確性が最重要のもの） | 華道家・流派との協力。撮影同意・利用範囲の書面化（Sonnet 4.6 がドラフト、Junt 確認）。クレジット表記 |
| ② ストック | 汎用一枝もの（菊・菖蒲・桃・南天等） | 公開Webアプリでの利用可否をライセンス条項で確認してから購入 |
| ③ AI生成 | 撮影時期を逃した季節花・希少花材（蝋梅・桂花・仏手柑等）の補完 | (a) 生成ツールの商用・公開利用規約を確認 (b) **Junt による様式・植物学的考証チェック通過が採用条件** (c) アプリ内クレジットに生成利用の明記方針を検討 |

## 12. 変更履歴 v1.1（2026-07-10 追記）— QR作成機能（メーカーモード）

### 追加された機能
- 同一ファイル内に学芸員用「メーカーモード」を追加。起動URL: `hana-ar.html?mode=maker`
- あらかじめ撮影した作品写真を読み込み（FileReader・端末外送信なし）、その上で花を選択・拡縮して構図を確認し、来館者向けQRを生成・PNG保存できる
- QRが運ぶ情報: `?work=作品ID&f=花ID&s=倍率`（写真自体はQRに入らない）
- 来館者側: プリセット付きQRを読むと、作品情報画面に「この作品には「◯」がえらばれています」と表示され、カメラ起動時にその花が指定倍率で自動配置される（位置は来館者が微調整。構図のピクセル再現はPhase 2トラッキングの領分）
- 不適合な花ID・存在しない花IDのプリセットは拒否しトースト表示

### 技術
- QRエンコーダを依存ゼロでJS内蔵(バイトモード/ECCレベルM/バージョン1-10自動/マスク0固定/v7以上のバージョン情報ブロック対応)
- 検証: v4〜v10の11ケース＋ビルド後HTML抽出版でOpenCVデコーダとの往復一致を確認済み
- 回帰テスト bt16〜bt21 を追加（QRエンコーダ、プリセット適用/拒否、メーカーモード初期化、単一花置換）。全21件グリーン

### 実機検証チェックリストへの追加項目
- [ ] `?mode=maker` で写真読込→花選択→QR作成→PNG保存が動く（iOS/Android/PC）
- [ ] メーカーモードで保存したQRを実機で読み、指定した花が自動配置されて開く
- [ ] 不適合プリセットQR（手打ちURL）でトースト表示され通常フローに戻る

## 13. 変更履歴 v1.2（2026-07-12 追記）— Playwright自動回帰テスト整備

### 実行結果（実際に実行した結果。3回連続実行して安定性を確認）
Chromiumの `--use-fake-device-for-media-stream` で疑似カメラを供給し、来館者フロー・
メーカーモード・器適合ロジック・タッチ操作・合成保存・オフライン動作までを自動テストで検証。
**18件中18件グリーン**。

| ID | 内容 |
|---|---|
| bt01–02 | 作品ID解決（正常系・異常系） |
| bt03 | カメラ起動・video要素への映像到達 |
| bt04–06 | 器適合フィルタ（narrow/medium/wide） |
| bt07–10 | 花の配置・ドラッグ移動・複数配置・消去モーダル |
| bt11 | 合成保存（ダウンロード発生・ファイル名検証） |
| bt12 | 外部リクエスト無し（オフライン原則） |
| bt13 | オフライン状態での動作 |
| bt14–16 | メーカーモード（作品/花選択肢・QR生成・URLパラメータ） |
| bt17–18 | プリセットQRの適用・不適合プリセットの拒否 |

### テスト整備中に発見・修正した不具合（テストコード側）
1. `page.waitForFunction(fn, options)` の引数省略により、指定したタイムアウトが効かずデフォルト30秒になっていた（`fn, undefined, options` が正しい形）
2. Chromium起動引数 `--use-file-for-fake-video-capture=`（空値）が疑似カメラ映像取得を不安定化させていた。削除して解消
3. `context.setOffline(true)` 後、アサーション失敗時に `false` へ戻さないと後続テストがオフライン汚染される問題。try/finallyで保護
4. bt09の当初実装は器適合と季節タブの既定表示仕様を誤解していた（既定は選択中の季節のみ表示）。季節タブ切替を組み込んで修正
アプリ本体(hana-ar.html)側の不具合は本テスト整備では検出されなかった。

### 構成
- `tests/hana-ar.spec.js`: 自己完結型（Node標準httpモジュールで静的配信、外部npm依存はplaywrightのみ）
- `.github/workflows/test.yml`: `workflow_dispatch` トリガーのみ（キャプション工房・スライドブラッシュアップ工房と同じ運用方式）
- `package.json`: playwright依存を宣言

### GitHub側で行う作業（Junt対応）
1. `hana-ar` リポジトリのルートに `package.json` をアップロード
2. `tests/hana-ar.spec.js` を作成（GitHub「Create new file」でファイル名欄に `tests/hana-ar.spec.js` と入力するとフォルダが自動生成される）
3. `.github/workflows/test.yml` も同様に「Create new file」でファイル名欄に `.github/workflows/test.yml` と入力して作成（隠しフォルダの心配なし）
4. リポジトリの「Actions」タブ →「hana-ar Playwright回帰テスト」→「Run workflow」で手動実行
5. 実行結果はActionsのログで確認。将来コードを変更するたびに同じ手順で再実行できる

### パイプライン（make_flower_asset.py）の高精度化
切り抜きの甘さ（フチの灰色残り・ギザつき・内部の背景ポケット）を最小化するため、numpy/scipy を用いた
高精度マッティングを既定にした（未導入環境は `--legacy` で旧方式にフォールバック）。
1. **アンチエイリアスなソフトアルファ**：境界画素のアルファを背景色との距離で連続化しギザつきを除去
2. **エッジ除染（despill）**：半透明画素から背景色の混色分を差し引き、灰色のフチ残りを消す
3. **背景色マップ**：前景を最近傍背景色で埋めた基準でグラデ背景にも強い
4. **微小ゴミ除去・小穴埋め**：切り抜き残りの孤立塊を除去、被写体内部の小穴を補間
5. **`--global`**：連結性を使わず色だけで背景判定（**葉に囲まれ外周とつながらない内部の背景ポケット**も除去）。
   被写体に淡色・無彩色部分が無い花材専用（桂花・石榴花で使用）。淡色の牡丹は連結フラッドのまま許容差を上げて対応。

## 22. 変更履歴 v2.1（2026-07-16 追記）— iPhone操作性の改善（スライダー拡縮・二本指回転・縮尺統一）

### 背景
iPhone実機で「二本指操作が難しい」「花材のデフォルトが大きすぎる（蘭以外）」との指摘。ピンチ拡縮の廃止と表示縮尺の正規化で対応した。

### 変更点（hana-ar.html）
1. **拡大縮小をスライダー化**：カメラ画面下部とメーカー画面に大きさスライダー（0〜100%）を追加。0〜100%が倍率0.25〜2.0に線形対応（`SMIN/SMAX`）。指1本で扱えるようにした。つまみは28pxでタッチしやすい。
2. **二本指は回転のみ**：従来の「二本指で拡縮＋回転」から拡縮を分離し、二本指は回転専用（`mode:"rotate"`）に。角度差のみを反映するため滑らかで、意図しない拡縮が起きない。
3. **表示縮尺の統一**：全花材を「幅220px固定」で表示していたため、縦長画像（桃214×1200等）が極端に大きく表示されていた。これを**高さ340px基準**に正規化（`DISP_H`＋`dispDims()`、CSS `.bloom img`）。基準は従来ちょうど良かった蘭の表示高さに合わせた。全花材が同じ基準高さで表示され縮尺がそろう。
4. デコード完了後に`applyTransform`を再適用し、アンカー（留め）位置のずれを防止。

### 同期
- 花を選択／配置／プリセット適用時にスライダー位置を現在倍率へ同期（`syncSizeSlider`）。
- メーカーモードは`makerBloom`を専用スライダー（`#mkSizeSlider`）で操作。マウス環境の学芸員でも大きさ設定が可能。

### QRプリセットへの影響
倍率`s`の意味（表示サイズへの乗数）は不変だが、基準表示サイズが幅220→高さ340に変わったため、**旧基準で発行済みのQRの見え方は変わる**。未公開のため実害なし。今後発行するQRは新基準。

### GitHub Pages
ルート(`/hana-ar/`)がindex.html不在で404になっていたため、クエリ保持でhana-ar.htmlへ転送する`index.html`を追加。

### テスト
Playwright 18件グリーン維持（実行確認）。追加でスクリプト検証：春4種すべて表示高さ340px・scale=1で統一、スライダー80%→倍率1.65を確認。

## 23. 変更履歴 v2.2（2026-07-16 追記）— ビジターモードUI改善（左レール化・個別削除・複数配置の変化・回転スライダー・写真保存）

iPhone実機からの5点の指摘に対応した。

1. **花材選択UIを左レールへ移動**：季節タブ・様式タブ・花材サムネイルを画面下部の横並びから、左端の縦レール（幅96px・縦スクロール）へ移した。カメラ映像の中央が選択UIに塞がれなくなった。下部は「大きさ・回転スライダー＋個別削除＋シャッター」の薄い帯のみ。
2. **配置した花の個別削除**：下部バー右端に赤い✕ボタンを追加。花を選択すると活性化し、タップでその1本だけを削除する。🗑（全消し・確認モーダル付き）は従来どおり。
3. **同一花材の複数配置に変化**：同じ花を2本目以降配置すると、わずかな軸回転（±6°〜）・左右反転（アンカー基準）・縮小（最大-15%）・位置ずらしを自動適用し、複製に見えないようにした（placeFlower内・決定論的）。左右反転は`state.flip`としてcanvas合成にも反映。
4. **回転もスライダー化**：「回転」スライダー（-90°〜+90°）を大きさスライダーの下に追加。二本指回転も残置（操作するとスライダーへ同期）。
5. **写真フォルダへの保存**：シャッター後、Web Share API（`navigator.share`＋files）対応端末では共有シートを開く。iPhoneでは「画像を保存」で写真アプリに保存できる。非対応環境は従来のダウンロードにフォールバック（テストbt11はこの経路で不変）。

その他: 操作ヒントの位置を左レールと重ならない右側領域へ調整。`setPointerCapture`を try-catch で保護（合成イベント対策）。テスト18件グリーン維持（実行確認）。メーカーモードのUIは変更なし。
